# Галич Агро — сайт + окрема адмін-панель

Монорепозиторій з двома незалежними Next.js застосунками:

- **`public-site/`** — публічний односторінковий сайт. Рендериться на сервері
  (SSR), вміст читає з бази даних, віддає прямий HTML для пошукових систем
  (хороше SEO). Тут же живе весь API (`/api/content`, `/api/admin/*`).
- **`admin-panel/`** — окрема адмінка. Розгортається як **окремий проєкт
  Vercel на іншому домені/піддомені**, ніде не посилається публічно, і
  спілкується з `public-site` лише через API з реальною автентифікацією.

Дані (тексти, SEO, зображення) зберігаються в PostgreSQL, зображення — у
Vercel Blob. Це вже не клієнтське сховище в браузері — сайт лишається
однаковим для всіх відвідувачів і переживає закриття вкладки чи оновлення
сторінки.

## Модель безпеки коротко

1. Пароль адміна ніколи не зберігається у відкритому вигляді — тільки
   bcrypt-хеш у змінній середовища `ADMIN_PASSWORD_HASH`.
2. Після входу видається JWT-сесія (2 години), у httpOnly + Secure cookie —
   недоступна для JavaScript, не читається через XSS.
3. Ендпоінти `/api/admin/*` приймають запити **лише** з точного домену
   адмінки (`ADMIN_ORIGIN`) — і через CORS, і через додаткову серверну
   перевірку заголовка `Origin`, що також закриває CSRF з чужих сайтів.
4. Невдалі спроби входу пишуться в базу і після 5 невдач за 15 хвилин з
   однієї IP — вхід тимчасово блокується.
5. Усі дані на вході валідуються (zod) з лімітами довжини; завантажені
   зображення перевіряються за типом (PNG/JPEG/WEBP/SVG/ICO) і розміром
   (до 4 МБ) перед збереженням у Blob.
6. Адмінка віддає заголовки `X-Robots-Tag: noindex`, `X-Frame-Options: DENY`
   — не індексується пошуковиками і не може бути вбудована в чужий iframe.
7. HTTPS на обох доменах — автоматично від Vercel.

## Крок 1. Створення репозиторію на GitHub

```bash
cd galych-agro
git init
git add .
git commit -m "Initial commit: public site + admin panel"
gh repo create galych-agro --private --source=. --remote=origin --push
```

Якщо немає `gh` (GitHub CLI) — створіть порожній приватний репозиторій
вручну на github.com, потім:

```bash
git remote add origin https://github.com/<ваш-акаунт>/galych-agro.git
git branch -M main
git push -u origin main
```

## Крок 2. База даних (Postgres)

Підійде будь-який Postgres. Найпростіше — прямо з дашборду Vercel:

1. Vercel → Storage → Create Database → Postgres (Neon) → створити.
2. Виконайте вміст `db/schema.sql` у цій базі один раз (через вкладку
   "Query" у дашборді Neon/Vercel або через `psql`).
3. Скопіюйте видану змінну `POSTGRES_URL` — вона знадобиться в кроці 4.

## Крок 3. Сховище зображень (Vercel Blob)

Vercel → Storage → Create Database → Blob → створити. Токен
`BLOB_READ_WRITE_TOKEN` з'явиться автоматично при підключенні до проєкту
`public-site` (крок 4).

## Крок 4. Деплой `public-site` на Vercel

1. Vercel → Add New → Project → виберіть репозиторій `galych-agro`.
2. **Root Directory**: `public-site`.
3. Framework Preset: Next.js (визначиться автоматично).
4. Підключіть щойно створені Postgres і Blob сховища до цього проєкту
   (Settings → Storage), або вручну додайте змінні середовища:

   | Змінна | Значення |
   |---|---|
   | `POSTGRES_URL` | зі кроку 2 |
   | `BLOB_READ_WRITE_TOKEN` | зі кроку 3 |
   | `JWT_SECRET` | результат `openssl rand -base64 48` |
   | `ADMIN_PASSWORD_HASH` | результат кроку 6 нижче |
   | `ADMIN_ORIGIN` | URL адмінки, наприклад `https://galych-agro-admin.vercel.app` (без `/` в кінці) |
   | `NEXT_PUBLIC_ADMIN_URL` | той самий URL адмінки, або залиште порожнім, щоб сховати посилання у футері |

5. Deploy. Запишіть видану адресу, наприклад `https://galych-agro.vercel.app`
   — вона знадобиться адмінці.

## Крок 5. Деплой `admin-panel` на Vercel (окремий проєкт!)

1. Vercel → Add New → Project → **той самий репозиторій** `galych-agro` ще
   раз (Vercel дозволяє підключати один репозиторій до кількох проєктів).
2. **Root Directory**: `admin-panel`.
3. Змінна середовища:

   | Змінна | Значення |
   |---|---|
   | `NEXT_PUBLIC_API_BASE_URL` | URL з кроку 4, наприклад `https://galych-agro.vercel.app` |

4. Deploy. Отримаєте окрему адресу, наприклад
   `https://galych-agro-admin.vercel.app` — саме її треба було вказати як
   `ADMIN_ORIGIN` у кроці 4. Якщо вказали заздалегідь неточно — поверніться в
   `public-site` → Settings → Environment Variables, виправте `ADMIN_ORIGIN`
   на фактичну адресу і натисніть Redeploy.

Після цього два проєкти "знають" одне про одного лише через ці дві змінні —
без спільного домену чи спільного коду виконання.

## Крок 6. Створення пароля адміна

Локально, у корені репозиторію:

```bash
npm install
npm run hash-password -- "Ваш-Надійний-Пароль-2024"
```

Скрипт виведе рядок `ADMIN_PASSWORD_HASH=$2b$...` — вставте це значення в
змінні середовища проєкту `public-site` на Vercel (крок 4) і натисніть
Redeploy. Сам пароль ніде, крім вашої голови, зберігати не потрібно.

Щоб змінити пароль пізніше — просто повторіть цю команду з новим паролем і
оновіть змінну середовища.

## Локальна розробка

```bash
npm install
cp public-site/.env.example public-site/.env.local   # заповніть значення
cp admin-panel/.env.example admin-panel/.env.local   # заповніть значення
npm run dev:site    # http://localhost:3000
npm run dev:admin   # http://localhost:3001
```

Для локальної розробки в `ADMIN_ORIGIN` вкажіть `http://localhost:3001`, а
в `NEXT_PUBLIC_API_BASE_URL` — `http://localhost:3000`.

## Коли буде свій домен

Просто прив'яжіть куплений домен до проєкту `public-site` у Vercel (Settings
→ Domains), а для адмінки за бажанням — окремий піддомен на іншому,
непублічному записі (наприклад, DNS-запис, який ви нікому не показуєте) або
лишіть її на `*.vercel.app` — вона й так не проіндексована та не пов'язана
публічним посиланням. Після зміни домену публічного сайту не забудьте
оновити `NEXT_PUBLIC_ADMIN_URL`, якщо хочете лишити приховане посилання у
футері, і `ADMIN_ORIGIN`/`NEXT_PUBLIC_API_BASE_URL`, якщо зміниться сама
адмінка.

## Що ще варто зробити для продакшн-рівня

- Увімкнути у Vercel двофакторну автентифікацію для акаунта, що керує обома
  проєктами.
- Раз на кілька місяців ротувати `JWT_SECRET` і пароль адміна.
- За потреби додати другого адміністратора — поточна модель розрахована на
  один спільний пароль; для кількох окремих облікових записів знадобиться
  таблиця користувачів замість одного `ADMIN_PASSWORD_HASH`.
